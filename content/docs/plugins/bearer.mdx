## Overview

The Bearer plugin provides Bearer token authentication by validating JWT access tokens from the `Authorization` header. It extracts tokens from the `Authorization: Bearer <token>` header format and validates them using the JWT plugin service. It supports both required and optional authentication modes and implements cooperative authentication.

### Features

- **Authorization Header Validation** — Extracts and validates `Authorization: Bearer <token>` headers or a configured header.
- **JWT Integration** — Uses the JWT plugin service for token validation (signature, expiration, blacklist).
- **Dual Authentication Modes** — Implements required authorization (`bearer.auth`, returns 401) and optional authorization (`bearer.auth.optional`, silent continuation).
- **Cooperative Authentication** — Works alongside other auth plugins by skipping validation if a `UserID` is already set.
- **Configurable Header** — Allows customization of the header name (defaults to `Authorization`).
- **Stateless Validation** — Does not require any server-side session storage.

## Configuration

### Plugin Configuration (`config.go`)

The plugin is configured via the following settings:

```go
type BearerPluginConfig struct {
	Enabled    bool   `json:"enabled" toml:"enabled"`
	HeaderName string `json:"header_name" toml:"header_name"`
}

func (config *BearerPluginConfig) ApplyDefaults() {
	if config.HeaderName == "" {
		config.HeaderName = "Authorization"
	}
}
```

### Standalone Configuration (TOML)

```toml
[plugins.bearer]
enabled = true
header_name = "Authorization"
```

### Library Mode Initialization (Go)

```go
import (
	bearerplugin "github.com/Authula/authula/plugins/bearer"
)

bearerplugin.New(bearerplugin.BearerPluginConfig{
	Enabled:    true,
	HeaderName: "Authorization",
}),
```

## API Reference (Hooks)

The Bearer plugin operates as a middleware plugin that validates incoming requests via hooks:

- **`bearer.auth`** — **Required authentication hook** that validates Bearer tokens. It returns an HTTP `401 Unauthorized` response if the token is invalid or missing, halting the request.
- **`bearer.auth.optional`** — **Optional authentication hook** that validates Bearer tokens if present. It silently continues if the token is invalid or missing, allowing the request to continue.

### Authentication Flow

The validation flow in the hooks is as follows:

1.  **Header Extraction** — The plugin extracts the specified header (default: `Authorization`) from the request and parses the `Bearer <token>` format.
2.  **Token Validation** — The extracted token is validated using the JWT plugin service, which verifies the signature, expiration, and blacklist status.
3.  **Success Path** — Valid tokens result in the `UserID` being set in the request context, allowing the request to proceed.
4.  **Failure Path** — Invalid or expired tokens result in a `401 Unauthorized` response with an appropriate error message for the required hook, or silent continuation for the optional hook.

### Client Request Format

Clients must send JWT access tokens in the configured header using the Bearer scheme:

```http
Authorization: Bearer eyJhbGciOiJFZERTQSJ9...
```

### Cooperative Authentication

The Bearer plugin implements cooperative authentication. If a request already has a `UserID` set by another authentication plugin (e.g., Session), the Bearer plugin will skip its validation, allowing multiple auth methods to work together seamlessly.

## Dependencies

- **JWT Plugin** — Required: the Bearer plugin validates tokens using the JWT service.
- **JWT Service** — Must be registered in the service registry by the JWT plugin.
